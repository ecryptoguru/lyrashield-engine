LyraShield Engine is the sandboxed repository-analysis process behind the LyraShield AI worker. It combines model-guided discovery with security tooling and emits bounded structured candidates for the application to normalize, correlate, and verify.

<Warning>
The engine is not an independent source of verification. Model confidence and generated proof-of-concept text never make a finding independently verified. The application owns evidence state, deterministic scanners, retests, and assurance reports.
</Warning>

## Supported boundary

- Production invokes the `lyrashield` executable in non-interactive mode.
- Only approved GPT-5.6 Sol, Terra, or Luna deployments are accepted.
- OpenAI- and Azure-compatible endpoints are supported; other inherited provider paths are rejected.
- The application sends repository targets to the engine. URL targets use the application's pinned deterministic URL scanner.
- Inherited telemetry is forced off by the adapter.

## Execution flow

1. The worker resolves model, reasoning effort, budget, run identity, sandbox image, and target workspace.
2. The engine validates configuration before sandbox setup.
3. The sandbox and agents inspect the authorized repository within bounded context, output, agent, and spend limits.
4. The engine writes `run.json` and `vulnerabilities.json` under `strix_runs/<run-name>/`.
5. The worker schema-validates bounded artifacts, stores private usage checkpoints, and passes candidates through deterministic identity and evidence-state rules.

## Result-quality boundary

Repository tests, type checks, packaging, sandbox smoke, and worker compatibility prove implementation properties. They do not establish accuracy or coverage. The inherited upstream XBEN result is not LyraShield evidence; see the repository's `benchmarks/README.md` for the required evaluation corpus.

Run only against targets you own or are explicitly authorized to test.
